# 검색 기능명세 — 키워드 조회 · 0건 제보 루프

> 설계 근거·기각 대안은 [ADR-0011](../adr/0011-search.md). 제품 정의는 [PRD](../prd.md), 첫 화면 레이아웃은 [ADR-0008](../adr/0008-timeline-layout.md), 게임 엔티티는 [ADR-0007](../adr/0007-game-entity.md).
> 범위: 키워드 검색 = **M1(발견)**, 0건 제보 루프 = **M2(공급, 제보 폼 의존)**.

## 목표

- 필터(browse)로 못 푸는 **특정 대상 조회(lookup)** 제공 — "이 게임 지금 이벤트 있나?".
- 검색 0건을 **제보 유입 채널**로 전환 — 유저가 찾는데 없는 순간 = 제보 의사 최고점.

## 범위

1. 키워드 검색 (M1)
2. 0건 → 제보 유도 루프 (M2)

---

## 1. 키워드 검색 (M1)

**사용자 스토리**
- 방문자는 게임명(한글명·원제)·제작사로 이벤트를 검색해 원하는 항목만 본다.
- 검색어는 기존 필터(유형·상태·지역·플랫폼)와 **AND 결합** — 예: "카탄" + 선주문 + 진행중.

**엔드포인트**
- 기존 `GET /api/events` 에 `q`(선택) 파라미터 추가. 별도 라우트 없음.
- `q` 공백/빈 문자열이면 필터 미적용(기존 동작 유지).

**매칭 규칙**
- 대상: `Event.title`, `Game.titleKo`, `Game.titleOriginal`, `publisher`.
- `description`·`platform` 제외([ADR-0011](../adr/0011-search.md)).
- 대소문자 무시 부분일치(`lower(field) LIKE lower('%q%')`), 대상 필드 OR.
- 공개 피드 게이트(`moderationStatus = PUBLISHED`), 기간 그룹·예고 섹션 분리, 마감임박순 정렬은 그대로.

**구현 메모**
- `Event.gameId`가 JPA 연관이 아니라 단순 컬럼이므로 **2단계 질의**(Game 제목 매칭 → gameId 집합 → `Event.gameId IN`). 질의 횟수는 **고정 2회** 유지(N+1 금지).
- `EventSpecifications.keyword(q, gameIds)` 형태로 기존 `and` 체인 합류. 컨트롤러는 `@RequestParam(required=false) q: String?` 추가.
- `%`/`_` 등 LIKE 와일드카드 이스케이프 처리(사용자 입력이 패턴으로 오작동 방지).

## 2. 0건 → 제보 유도 루프 (M2)

**사용자 스토리**
- 검색 결과 0건이면 "이 이벤트를 알고 계신가요? 제보하기" 노출, 클릭 시 제보 폼(S4)에 **검색어 프리필**.

**규칙**
- 프리필은 프런트에서 처리(검색어를 제보 폼 라우트 상태로 전달). 백엔드 변경 불필요.
- 제보 폼(S4)은 M2 도입 → 그전까지 0건 화면은 안내 문구만.
- 남용 방지는 기존 제보 경로와 동일([ADR-0006](../adr/0006-abuse-prevention.md)).

**지표 연결**
- **검색 0건 → 제보 전환율** 을 공급 건강도 지표로 추적([product.md](../product.md)).

## 스코프 밖 (의도적)

- **검색어 로그·집계**: 운영자 시딩 수요 신호로 유용하나 테이블·집계 필요 → M2 이후 defer([ADR-0011](../adr/0011-search.md)).
- **자동완성·오타 교정·동의어**: 데이터 수백 건에선 무의미. 검색 지연/불만 관측 시 재검토.
- **FTS·검색 전용 저장소**: 수천 건대 성장 + 실측 지연 시([ADR-0011](../adr/0011-search.md) 에스컬레이션 조건).
