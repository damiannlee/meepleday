# 0009. URL 자동 채움을 자동 수집과 분리해 취급

- 상태: Accepted
- 맥락: [product.md](../product.md)는 플랫폼 자동 수집(크롤링)을 법적 리스크로 판단해 defer했음 — 텀블벅·Kickstarter는 ToS가 자동화된 수단을 명시 금지. 그런데 [PRD §4.3](../prd.md#43-등록-자동-채움)은 운영자·제보자가 붙여넣은 URL을 서버가 페치해 폼을 채우는 기능을 도입함. **둘 다 "서버가 남의 사이트에 요청을 보낸다"는 점은 같아서**, 같은 사안으로 묶으면 자동 채움까지 봉쇄되고 운영 병목이 방치됨.

## 결정

- **URL 자동 채움과 자동 수집을 별개 사안으로 취급한다.** 자동 수집의 defer 판단이 자동 채움을 막지 않음.
- 자동 채움의 허용 범위를 다음으로 한정:
  - **사용자가 명시적으로 제출한 URL 1건**에 대해서만 요청. 링크를 따라가며 추가 수집하지 않음.
  - 제출 1건당 **1회 페치**. 스케줄·반복·큐잉 없음.
  - 추출 대상은 **공개 OG 메타 태그**(`og:title`, `og:image`, `og:description`, `og:url`)만. 본문 파싱·플랫폼 전용 파서를 만들지 않음.
  - 원본 콘텐츠를 **저장하지 않음** — 이미지는 URL 핫링크, 텍스트는 운영자가 검토·수정하는 초안일 뿐.
  - 접근 통제(로그인·anti-bot)를 **우회하지 않음.** 차단되면 그대로 실패시키고 사람이 수기 입력.

## 근거

- **행위의 성격이 다르다.** 자동 수집은 우리 판단으로 반복·대량 요청을 보내는 것이고, 자동 채움은 사용자가 지금 보고 있는 페이지 1건을 대신 열어보는 것 — 사용자 행위의 대리 실행에 가까움. 브라우저·메신저의 링크 미리보기와 같은 계열.
- **서버 부하가 무시 가능.** 등록 건당 1회, 하루 수십 회 규모. ToS가 자동화된 수단을 금지하는 취지는 대량 수집으로 인한 부하·데이터 무단 축적인데, 여기엔 해당하지 않음.
- **데이터를 축적하지 않음.** 페치 결과는 폼 초안으로 소비되고 원본 텍스트가 그대로 저장되지도 않음.
- **운영 병목의 직접 해소.** 등록 소요 시간은 이 서비스가 조용히 죽는 가장 흔한 경로이고, 자동 채움은 그에 대한 가장 값싼 대응.

## 한계 — 이 결정이 보장하지 않는 것

- **법적 안전을 단정하지 않음.** ToS 문구는 "자동화된 수단"을 넓게 쓸 수 있고, 해석상 단건 페치까지 포함된다고 볼 여지가 남음. 이 ADR은 **위험도가 낮다는 판단**이지 적법 확인이 아님.
- 완화책: 접근 통제 우회 금지, 저장 금지, 요청량 최소 유지 — 위 세 가지가 지켜지는 한 실질 분쟁 가능성은 낮다고 봄. 플랫폼이 명시적으로 거부 의사를 밝히면(차단 응답 포함) **즉시 그 플랫폼에 대해 중단**.
- 자동 수집(M4+)으로 넘어갈 때는 이 ADR을 근거로 삼을 수 없음 — 별도 판단 필요.

## 기각한 대안

- **자동 채움도 함께 defer**: 일관돼 보이지만 성격이 다른 둘을 뭉뚱그린 것. 운영 병목을 방치해 서비스 지속성을 해침.
- **클라이언트(브라우저)에서 페치**: 서버가 요청을 안 보내니 책임을 회피하는 듯하나, CORS 때문에 대부분 실패하고 사용자 IP를 노출시킴. 기술적으로도 부적절.
- **플랫폼별 전용 파서 작성**: 추출 품질은 오르지만 각 사이트 구조에 의존하는 코드가 되어 "대량 수집 도구"에 훨씬 가까워짐 — 리스크를 스스로 키우는 선택.

## 보안 (필수)

서버가 임의 URL로 요청을 보내는 기능이므로 SSRF·자원 고갈 방어가 전제 조건 — [ADR-0006](0006-abuse-prevention.md) 개정 참조.
