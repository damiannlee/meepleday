# 0010. 성장 목표는 명시하되 구현하지 않는다

- 상태: Accepted
- 맥락: [PRD](../prd.md) 확정 과정에서 규모 목표를 "중간 목표(성장 대비) — 누적 이벤트 수만 건, 동시 접속 수백 명"으로 잡음. 그대로 받으면 피드 캐시 레이어·이미지 프록시·읽기 복제를 지금 만들게 되는데, 현재 트래픽은 0이고 관측된 성능 문제도 없음.

## 결정

- **수치 목표는 문서에 명시하되, 대응 인프라는 지금 만들지 않는다.**
  - 목표: 누적 이벤트 수만 건, 동시 접속 수백 명.
  - 응답 예산: 피드 p95 ≤ 500ms, 상세 p95 ≤ 300ms.
- **지금 하는 것**: 인덱스 설계, N+1 제거, 페이징. 즉 **비용이 거의 없고 나중에 하면 비싼 것들**.
- **지금 하지 않는 것**: 피드 캐시 레이어, 이미지 프록시/리사이즈, 읽기 복제본, CDN 오리진 캐시 튜닝.
- **아키텍처에는 삽입 지점만 표시**해 둔다 — 캐시가 들어갈 자리를 문서·코드 구조상 알아볼 수 있게 하되 실제 구현은 비움.
- **도입 트리거**: 응답 예산 초과가 실제로 관측될 때. 추측이 아니라 측정이 근거.

## 근거

- **검증되지 않은 트래픽에 대한 선제 최적화는 복잡도만 남긴다.** 캐시는 무효화 버그·스테일 데이터·디버깅 난이도를 동반하는데, 그 비용을 지불할 이유(관측된 지연)가 아직 없음.
- 응답 예산을 먼저 정의해두면 **"언제 만들지"가 자동으로 결정됨** — 임의 판단이 아니라 측정 기반 트리거.
- 반대로 인덱스·N+1·페이징을 미루면 데이터가 쌓인 뒤 고치기 비싸짐 → 이 셋만 선제 처리하는 게 비용/편익이 맞음.
- 포트폴리오 관점에서도 **"성장을 고려했으나 근거 없이 만들지 않았다"** 가 "일단 캐시를 붙였다"보다 방어 가능한 서사. 판단의 근거와 트리거를 남기는 것이 요점.

## 기각한 대안

- **중간 목표에 맞춰 지금 구축**: 성장에 대비돼 보이지만, 이 규모(운영자 1명·주 20건 등록)에서 캐시·프록시는 순수 부채. 관측 없는 최적화.
- **소규모 현실선으로 목표 자체를 낮춤**(수천 건·동시 수십 명): 정직하지만 목표를 낮게 박으면 나중에 확장 판단의 기준점이 사라짐. **목표와 구현 시점을 분리**하는 편이 낫다고 판단.
- **목표를 아예 문서화하지 않음**: 가장 가볍지만 "성능을 생각 안 했다"와 구분되지 않음.

## 트레이드오프

- 성능 저하가 관측되는 시점에 대응 작업이 몰림 → 응답 예산을 **측정할 수단**(관측성, [PRD §6.6](../prd.md#66-관측성))이 먼저 있어야 이 결정이 성립. 측정 없이 이 ADR을 따르면 그냥 방치가 됨.
- 트래픽이 급증하면 대응 리드타임 동안 사용자 경험이 나빠질 수 있음. 이 서비스의 성장 곡선상 급증 가능성은 낮다고 봄(커뮤니티 공유 기반, 시즌성 없음).
