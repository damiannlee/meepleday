# 0003. 인증을 마지막 개발 단계로 지연

- 상태: Accepted
- 맥락: 출시 시엔 회원가입·로그인이 있는 공개 서비스. 다만 인증을 **먼저** 붙이면 개발·테스트마다 로그인 벽에 부딪혀 비효율. 또한 **공개 배포 자체를 Phase 2 완료 후 완성형으로** 진행([ADR-0005](0005-deployment-target.md)) → Phase 1은 로컬 개발 마일스톤으로만 존재하고 대중에 공개되지 않음.

## 결정

- 인증(Spring Security)·북마크·이메일 알림을 **Phase 2(마지막 단계)** 로 배치.
- Phase 1은 로그인 없이 조회·제보·검수까지 완결.
- 제보자는 당장 `User` FK가 아니라 `submitterName`/`submitterEmail` 텍스트로 수집 → 인증 도입 시 User FK로 승격.
- 운영자 API는 `/api/admin/**` 로 **경로 분리**만 해 두어, 인증 도입 시 한 곳에서 잠글 수 있게 함.

## 근거

- **개발 순서 결정**: 로그인 벽을 마지막에 붙여 개발·테스트 내내 반복 로그인 비용 제거. 핵심 가치(한눈에 보기 + 크라우드소싱 큐레이션)를 먼저 완성해 응집도 높은 순서로 구현.
- 공개 배포가 Phase 2 이후라 **출시 첫인상이 완성형**(인증·북마크·알림 포함) → 반쪽짜리 무인증 피드를 대중에 노출하지 않음.
- (초안의 "무인증으로 먼저 공개 검증" 근거는 폐기 — 배포가 Phase 2 이후라 Phase 1이 공개되지 않으므로 성립 안 함.)

## 기각한 대안

- **인증부터 구현**: 정공법이나 핵심 가치 검증이 인증 뒤로 밀리고, 개발 내내 로그인 오버헤드. 스코프 관리상 후순위.

## 지연에 따른 부채(의도적)

- `/api/admin/**` 무인증은 **로컬 개발 한정** — 공개 배포가 인증 완비 후라([ADR-0005](0005-deployment-target.md)) **대중 노출 창이 없음** → 부채 위험 낮음. 단 배포 전 잠금 완료는 필수.
- 제보자 텍스트 필드(`submitterName`/`submitterEmail`)는 인증 도입 시 User FK로 마이그레이션 필요.
