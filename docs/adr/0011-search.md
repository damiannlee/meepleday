# 0011. 키워드 검색 — LIKE 부분일치로 시작

- 상태: Accepted
- 맥락: 첫 화면은 기간 그룹 타임라인 + 유형·상태 필터([ADR-0008](0008-timeline-layout.md), [PRD §5.3](../prd.md#53-필터정렬))로, 전부 **browse(둘러보기)** 도구. 그러나 "아컴호러 한국어판 지금 선주문 하나?" 같은 **특정 대상 조회(lookup)** 수단이 없음. [PRD §4.6](../prd.md#46-인증과-로그인-벽)이 "검색·필터"를 무인증 기능으로 전제하나 구현 방식은 미정 → 이 ADR에서 확정.
- 데이터 규모: 활성 20~30건 유지 목표, 누적도 수백 건대([product.md](../product.md)). 자동 수집(M4+) 전까지 대량화 없음.

## 결정

- 타임라인 상단 필터 영역에 **검색어 입력 1개**. 별도 검색 페이지 없이 기존 `GET /api/events` 에 `q` 파라미터 합류.
- 매칭 대상: `Event.title` + **`Game.titleKo` / `Game.titleOriginal` 양쪽** + `publisher`.
  - 해외 이벤트는 원제로 저장되므로([PRD §5.6](../prd.md#56-표기-언어)) 한글명·원제 어느 쪽으로 쳐도 잡혀야 함.
  - **`description` 제외** — 길고 노이즈만 늘려 정밀도 저하.
  - **`platform` 제외** — 이미 2급 필터로 존재([ADR-0008](0008-timeline-layout.md)), 검색어로 중복 제공할 값어치 낮음.
- 구현: `LIKE '%q%'` **대소문자 무시 부분일치**, 양쪽 `lower()` 정규화 — `Ashes Reborn` 등 원제 때문에 필수. H2·PostgreSQL 양쪽 지원(이식성 규칙 준수).
- **M1(발견)** 범위. 0건 유도 루프만 제보 폼에 의존해 M2.

## 기각한 대안

- **PostgreSQL FTS(`tsvector`/`to_tsquery`)**: H2에 없어 이식성 규칙 위반. PG 기본 파서는 한국어 형태소 분석 불가 → `simple` 설정으론 "보드게임"↔"보드" 미매칭. 제대로 하려면 `pg_bigm`/mecab 계열 확장 의존 → RDS 확장 가용성부터 별건 확인 필요. 규모 대비 과함.
- **`pg_trgm` 인덱스**: 성능은 좋으나 PG 전용 확장이라 이식성 위반. 한글은 3-gram 특성상 2글자 검색어("카탄")가 잘 안 걸림.
- **Elasticsearch/OpenSearch**: 활성 20~30건에 검색 전용 인프라는 명백한 과설계(YAGNI). 운영비도 EC2 한 대 값. 자동 수집조차 M4+로 미룬 단계에서 부적절.
- **자동완성·오타 교정**: 데이터 수백 건에선 후보가 안 나옴. 검색 로그 축적 후 재검토.

## 근거

- `LIKE '%...%'`는 선행 와일드카드라 인덱스 미사용(seq scan)이나 **수백 건 규모에선 측정 불가능한 차이** → 지금 성능 최적화는 관측 전 선제 최적화(YAGNI). [ADR-0010](0010-growth-target-unbuilt.md)의 "성장 목표는 명시하되 구현하지 않는다"와 같은 태도.
- 이식성(H2↔PostgreSQL 단일 SQL) 규칙을 지키는 유일하게 단순한 선택. 인프라 추가 0.
- "지금 안 하는 이유 + 나중에 할 조건"을 명시해 에스컬레이션 시점을 문서화.

## 트레이드오프 / 구현 주의

- **`Event.gameId`는 JPA 연관이 아닌 단순 컬럼**([ADR-0007](0007-game-entity.md)) → Criteria join 불가. 게임 제목 매칭은 **2단계 질의**로 처리:
  1. `Game` 에서 제목 매칭 → `gameId` 집합 수집(질의 1회),
  2. `Event.title LIKE ... OR Event.gameId IN (:ids)` 로 피드 질의.
  - 행마다 질의하는 N+1이 아니라 **고정 2회**임을 유지할 것(프로젝트 N+1 금지 규칙).
- 게임 미연결 이벤트(`gameId = null`)는 `Event.title` 로만 매칭 — M1 시점엔 게임 매칭이 수작업이라 미연결이 다수일 수 있음.

## 에스컬레이션 조건 (관측 시 상향)

- 자동 수집(M4+) 이후 이벤트가 **수천 건대**로 커지고 **검색 지연이 실측**되면 → `pg_bigm`(한국어 부분일치 GIN 인덱스, RDS 가용 확인 후) 또는 검색 전용 저장소 검토.
- 검색 품질 불만(오타·동의어)이 정성 신호로 반복되면 → 자동완성·동의어 사전 재검토.

## 후속 (별건, 이 ADR 범위 밖)

- **검색 0건 → 제보 유도 루프**: 결과 없음 화면에서 검색어를 제보 폼에 프리필해 크라우드소싱 유입 채널화. 제보 폼이 M2라 그때 활성화. 명세 [spec/search.md](../spec/search.md), 지표 [product.md](../product.md).
- **검색어 로그**: 운영자 시딩 수요 신호(수요 있으나 공급 없는 항목). 테이블·집계 필요 → M2 이후 defer.
