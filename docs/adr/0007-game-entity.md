# 0007. 게임을 별도 엔티티로 두고 Event를 1:N 연결

- 상태: Accepted
- 맥락: 같은 게임이 여러 이벤트로 등장함 — 해외 펀딩 → 국내 정발 선주문 → 이후 특가. 서비스를 "이벤트 캘린더"로 재정의하면서([PRD §1](../prd.md#1-서비스-정체성)) 이 관계를 다룰지 결정해야 함.

## 결정

- **`Game` 1 : N `Event`** 도입.
- `Game` 필드: `id`, `titleKo`(nullable), `titleOriginal`, `publisher`(nullable), `bggId`(nullable), `createdAt`.
- **`Event.gameId`는 nullable** — 게임을 특정할 수 없는 이벤트가 실재함(종합 행사, 여러 게임 묶음 특가). 필수화 금지.
- **동일 게임 판정은 사람이 한다.** 등록·검수 화면이 제목 유사도 기반 **후보를 제시**하고, 확정 또는 "새 게임으로 등록"은 운영자가 선택.
- 게임 페이지(S3)에서 해당 게임의 이벤트를 타임라인으로 노출.

## 근거

- **장기 비전 2건의 선결 조건.** 게임 단위 생애주기 추적([PRD §8.2](../prd.md#82-게임-단위-생애주기-추적))과 가격 비교가 모두 `Game` 없이는 성립 불가. 나중에 도입하면 기존 이벤트 전량의 소급 매칭이 필요한 대공사.
- 같은 게임이 피드에 중복 노출되는 문제를 구조적으로 해결.
- "이거 국내 정발 나오나?"라는 실제 질문에 답할 수 있게 됨 — 타깃 페르소나의 핵심 불만 중 하나.
- `bggId`를 미리 열어둬 BoardGameGeek 연동 여지 확보(지금은 사용 안 함).

## 기각한 대안

- **묶지 않음 (이벤트가 단위)**: 구현·제보·검수가 가장 단순하고 현행 스키마 그대로. 그러나 같은 게임의 중복 노출을 못 막고, 장기 비전 2건이 봉쇄됨. 도입 시점이 늦을수록 비용이 커지는 종류의 결정이라 지금 치르는 편이 쌈.
- **게임명 문자열 태그(정규화된 문자열로 느슨하게 연결)**: 중간 비용으로 "이 게임의 다른 이벤트" 표시까지는 가능. 그러나 표기 흔들림(`글룸헤이븐 2판` / `Gloomhaven 2E` / `글룸헤이븐 세컨드 에디션`)을 해결 못 해 결국 정규화 주체가 필요해짐 → 엔티티로 수렴.
- **nullable `game_id` 컬럼만 미리 두고 미사용**: 마이그레이션 부담을 이연하지만, 쓰지 않는 컬럼을 두는 것 자체가 YAGNI 위반이고 실제 도입 시점의 작업량은 거의 그대로.
- **자동 매칭으로 확정(사람 개입 없음)**: 제목 유사도만으로는 확장팩·재판·한글판 구분이 불가능(`Gloomhaven` vs `Gloomhaven: Jaws of the Lion`). 오매칭이 생기면 게임 페이지가 오염되고 되돌리기 어려움 → 후보 제시까지만 자동화.

## 트레이드오프 / 비용

- **검수 플로우에 판정 단계가 추가됨.** 제보 1건을 처리할 때 "이 게임이 기존 Game과 같은가"를 사람이 판단해야 하므로 검수 시간이 늘어남 → 운영자 부담 지표([product.md](../product.md))로 관측 필요.
- 게임 데이터 자체의 품질 관리(중복 Game, 오탈자)가 새로운 운영 항목이 됨.
- 게임 페이지는 연결 이벤트가 쌓이기 전엔 보여줄 것이 거의 없음 → 초기엔 가치가 낮은 화면. 진입 조건은 [PRD §8.2](../prd.md#82-게임-단위-생애주기-추적).
- 타임라인·게임 페이지에서 `Event`와 `Game`을 함께 로드하므로 **N+1 위험 지점**. 배치 로딩 필수.
