# 0006. 익명 제보 남용 방지

- 상태: Accepted
- 맥락: `POST /api/events`는 인증 없이 열린 공개 쓰기 엔드포인트([ADR-0003](0003-auth-last.md)). 봇·스팸이 확실히 접근하는 표면. 다만 제보는 PENDING으로 생성돼 운영자 검수 전엔 공개 피드에 노출되지 않음([ADR-0002](0002-moderation-on-event.md)).

## 심각도 재평가

- **공개 피해는 검수 게이트가 이미 차단** — 봇이 스팸을 넣어도 PUBLISHED 되기 전엔 피드/상세에 안 뜸. 공개 defacement 위험은 낮음.
- 실제 위협은 **검수 큐 범람 + DB 팽창** → 운영자 검수 부담·저장 비용. 심각도 **중(medium)**, 치명(critical) 아님. 관측 전 과설계 경계 대상(YAGNI).

## 결정 (방어적 최소만 지금, 나머지 defer)

- **Honeypot 필드**: 제보 폼에 사람 눈에 안 보이는 필드 1개 추가. 값이 채워지면(=봇) 조용히 거절. 인프라 0, 비용 0.
- **IP 단위 rate limit**: 인메모리 토큰버킷(예: Bucket4j)으로 IP당 제보 상한(예: 5건/시간). 단일 인스턴스 전제.
- 초과·honeypot 적중은 `400`이 아니라 **성공처럼 응답 후 무저장**(봇에게 실패 신호 안 줌) 또는 조용한 429 — 구현 시 택.

## 명시적 defer (관측 시 도입)

- **CAPTCHA / Cloudflare Turnstile**: 무료·프라이버시 친화적, 호스팅 무관(위젯+서버검증)이라 AWS 배포([ADR-0005](0005-deployment-target.md))에도 그대로 적용. UX 마찰 있어 실제 봇 트래픽 관측 시 도입. AWS WAF는 고정비·과설계라 후순위.
- **분산 rate limit(Redis 등 공유 스토어)**: 백엔드 수평 확장 시에만 필요. 현재 단일 인스턴스라 불필요.
- **제보자 신뢰도·차단 목록**: 인증(Phase 2) 이후 User 단위로 재설계.

## 근거

- 검수 게이트가 1차 방어선이라는 사실이 심각도를 낮춤 → 무거운 방어(캡차)를 관측 전 강제할 근거 없음.
- Honeypot + IP 제한은 인프라 추가 없이 봇 대량 트래픽의 대부분을 걸러 큐 범람을 실용적으로 억제. 비용 대비 효과 최상.
- 보안 인식을 문서로 남기되 과설계는 회피 — 스코프 관리.

## 트레이드오프 / 후속 부채

- 인메모리 rate limit은 인스턴스 재시작 시 카운터 소실·다중 인스턴스 미공유 → 배포 확장 시 재검토(위 defer 항목).
- honeypot은 단순 봇엔 유효하나 표적 공격엔 무력 → Turnstile로 상향 경로 확보해 둠.
