## 1. 서비스 정체성

### 한 문장

> **흩어져 있는 보드게임 이벤트 일정을, 국내·해외 구분 없이 한 곳에서 보는 서비스.**

### 정체성 재정의 (중요)

- 종전 정의 = "보드게임 **펀딩·선주문·특가** 애그리게이터".
- 신규 정의 = "보드게임 **이벤트 캘린더**" — 보드게임과 관련된 이벤트라면 유형 불문 수용(펀딩·선주문·특가·오프라인 행사·아직 열리지 않은 예고).
- 축의 이동: **상거래(무엇을 사나)** → **일정(언제 무슨 일이 있나)**.
- 파급: 달성률·모금액 같은 상거래 수치의 우선순위 하락([§4.4](#44-데이터-신선도)), 오프라인 행사가 1급 시민으로 승격([§4.1](#41-이벤트-모델)).

**재정의의 리스크 (관측 대상)**
- 실제 피드의 대부분은 여전히 펀딩일 가능성 — 국내 오프라인 행사는 연간 수십 건 규모라, 유저 체감은 "펀딩 사이트"에 머물 수 있음.
- 행사는 **플랫폼이 없음** — 정보가 커뮤니티 공지·SNS에 흩어져 있고 OG 메타가 부실해 URL 자동 채움([§4.3](#43-등록-자동-채움))이 가장 안 먹히는 유형. 예고도 동일.
- → 재정의가 문서에만 남고 실물은 애그리게이터로 수렴할 위험. **행사·예고 게시 건수를 별도 지표로 추적**하고 M1 완료 기준에 포함([§3](#3-핵심-가치와-퍼널), [§7](#7-로드맵-전면-재수립)).

### 우리가 하는 일 / 하지 않는 일

| 한다 | 하지 않는다 |
|---|---|
| 이벤트 일정의 수집·정규화·게시 | 결제·펀딩 중개 (거래는 원본 플랫폼) |
| 유형·상태 기준의 탐색 | 상품 판매·중개 수수료 |
| 마감 추적과 알림 | 리뷰·평점·커뮤니티 (장기 비전, [§8](#8-장기-비전)) |
| 원본으로의 정확한 연결 | 원본 콘텐츠의 저장·재배포 ([§4.5](#45-이미지-정책)) |

## 2. 타깃 사용자

### 페르소나 (단일 주 타깃)

**"국내·해외 가리지 않는 보드게이머"**

- 국내 텀블벅 펀딩도, 해외 Gamefound 캠페인도, 오프라인 보드게임 행사도 관심 대상.
- 정보가 텀블벅·Kickstarter·Gamefound·보드라이프·각종 쇼핑몰·커뮤니티 공지에 흩어져 있어 **일정 파악에 반복 노동**이 듦.
- 놓치면 되돌릴 수 없는 것(펀딩 마감, 선주문 마감, 행사 예매)이 있어 **시간 압박**이 실재.

**핵심 불만(Pain)**
1. 어디를 봐야 하는지 모름 → 알았을 땐 이미 마감.
2. 플랫폼마다 흩어져 매번 여러 사이트 순회.
3. "언제까지인지"가 각 사이트마다 다른 형식으로 표기됨.

**기각한 타깃** — 근거 병기
- **보드게임 카페·샵 운영자(B2B)**: 재방문 동기는 가장 강하나 필요 정보(원가·유통사·물량)가 소비자용과 달라 사실상 다른 서비스. 지금 수용하면 화면이 둘로 쪼개짐.
- **해외 펀딩 헌터 전용**: 이미 정보력이 있어 우리 없이도 해결 가능 → 가치 증분이 작음.
- **국내 라이트 게이머 전용**: 이벤트 밀도가 낮아 "볼 게 없는" 상태 위험.

## 3. 핵심 가치와 퍼널

### 1차 가치 = **발견 → 추적** 퍼널

```
[진입]                     [전환]                    [리텐션]
커뮤니티 링크 클릭   →   타임라인에서 발견   →   북마크(로그인)   →   마감임박 알림   →   재방문
     무인증                    무인증                로그인 벽 ①              이메일
```

- **발견이 진입, 추적이 리텐션.** 발견 단계는 전부 무인증 — 공유 유입의 마찰을 0으로.
- 로그인 벽은 **단 한 곳(①)** — 북마크/알림 설정 시점. [§4.6](#46-인증과-로그인-벽) 참조.

### 지표 충돌 해소

- 종전 문제: 북극성 = 주간 재방문(추적형) vs 선행 = 원본 링크 클릭률(발견형) → 서로 다른 서비스를 가리킴.
- 해소: **퍼널 단계별로 지표를 계층화**하고 북극성은 리텐션에 둠.

| 단계 | 지표 | 의미 |
|---|---|---|
| 진입 | 공유 링크 유입 세션 수 | 커뮤니티 확산이 도는가 |
| 발견 | **원본 링크 클릭률** (세션당) | 방문이 실제 가치를 줬는가 |
| 전환 | 방문 → 북마크 전환율 | 발견이 추적으로 이어지는가 |
| 리텐션 | **주간 재방문 유저 (북극성)** | 도구로 자리잡았는가 |
| 리텐션 | 알림 클릭률 | 재방문 훅이 작동하는가 |

**공급 건강도 (별도 축)**
- 게시된 활성 이벤트 수 (국내/해외 비율) — 목표: 상시 활성 20~30건 이상.
- **행사·예고 게시 건수** — 정체성 재정의가 실물에 반영됐는지 검증하는 지표([§1](#1-서비스-정체성) 리스크). 0에 수렴하면 사실상 펀딩 애그리게이터.
- 제보 승인율 = PUBLISHED / (PUBLISHED + REJECTED).
- **운영자 등록 소요 시간 (건당)** — 신규 지표. [§4.3](#43-등록-자동-채움)의 효과 측정이자 서비스 생존 지표.

**초기 정성 목표**
- 실사용 보드게이머 5명이 "국내+해외+행사를 한 곳에서 보니 유용하다" 확인 → PMF 초기 신호.

## 4. 기능 요구사항

### 4.1 이벤트 모델

**결정: 단일 `Event` + 유형별 필드 분리, 한 피드에서 카드가 유형별로 변신.**

- 상위 `Event`가 공통 필드를 갖고, `EventType`에 따라 표시 필드가 달라짐.
- 기각 — **공통 필드만 표시**: 스키마는 단순하나 유형별 핵심 정보(행사의 장소 등)를 버려 정보 가치가 야윔.
- 기각 — **탭 분리(구매/행사)**: 각각은 최적화되나 "한 곳에서 본다"는 약속이 깨짐.
- 기각 — **행사를 후순위로 defer**: 타깃이 "이벤트라면 뭐든"을 원함 → 출시 커버리지가 정의와 모순.

**공통 필드**

| 필드 | 설명 |
|---|---|
| `title` | 이벤트 제목 (해외는 원제 유지, [§5.6](#56-표기-언어) 참조) |
| `type` | FUNDING / PREORDER / SALE / OFFLINE_EVENT |
| `region` | DOMESTIC / OVERSEAS |
| `platform` | 텀블벅·Gamefound·보드라이프·자체몰 등 (출처) |
| `sourceUrl` | 원본 링크 (모든 CTA의 목적지) |
| `imageUrl` | 원본 이미지 URL (핫링크, [§4.5](#45-이미지-정책)) |
| `startAt` / `endAt` | **둘 다 nullable** — 예고 이벤트 수용 |
| `scheduleNote` | 자유 입력 (`"2026년 4분기 예정"`, `"시기 미정"`) |
| `gameId` | Game FK, nullable ([§4.2](#42-game-엔티티)) |
| `moderationStatus` | PENDING / PUBLISHED / REJECTED |
| `submittedByUserId` | nullable (익명 제보 허용) |

**유형별 필드 (카드 변신의 근거)**

| 유형 | 추가 필드 | 카드에 표시 |
|---|---|---|
| FUNDING | 목표 달성 여부(bool, optional) | 플랫폼 · 마감 D-day |
| PREORDER | 예상 배송 시기, 가격 | 가격 · 마감 D-day |
| SALE | 할인가, 정가 | 할인율 · 종료 D-day |
| OFFLINE_EVENT | 장소, 주소, 참가비, 예매 링크 | 장소 · 개막 D-day |

- **달성률·모금액은 필드에서 제외.** 근거: 수동 등록 시점 스냅샷이라 즉시 낡고, 사용자 판단에 결정적이지 않다는 확인. 기존 Phase 1의 달성률 바는 제거 대상.
- 구현 노트: 유형별 필드는 JSON 서브타입 또는 nullable 컬럼 묶음. 유형이 4개로 고정적이므로 **nullable 컬럼 묶음이 1순위**(쿼리·검증 단순, 스키마 가시성 확보). JPA 상속(SINGLE_TABLE)은 유형 추가 시 이득이 생기면 재검토.

### 4.2 Game 엔티티

**결정: `Game` 1 : N `Event` 도입.**

- 같은 게임이 "해외 펀딩 → 국내 정발 선주문 → 특가"로 여러 이벤트를 가짐 → 이를 묶어 게임 단위로 조회 가능하게.
- 근거: 장기 비전 3개 중 2개(게임 생애주기 추적, 가격 비교)가 Game을 **선결 조건**으로 요구. 나중에 도입하면 대공사.
- 기각 — **묶지 않음(이벤트가 단위)**: 구현은 가장 싸지만 "이거 국내 정발 나오나?"에 답할 수 없고 피드에 같은 게임이 중복 노출.
- 기각 — **게임명 문자열 태그**: 중간 비용이지만 표기 흔들림(한글판/원제)을 못 잡음 → 결국 엔티티가 필요해짐.

**필드**: `id`, `titleKo`(nullable), `titleOriginal`, `publisher`(nullable), `bggId`(nullable, 장기 비전 대비), `createdAt`.

**대가 — 검수 플로우에 판정 단계 추가**
- 제보/등록 시 "이 게임이 기존 Game과 같은 것인가"를 사람이 판정해야 함 (`글룸헤이븐 2판` vs `Gloomhaven 2E`).
- 요구사항: 등록·검수 화면에 **게임 자동 매칭 후보 제시 + "새 게임으로 등록" 선택지**. 자동 매칭은 제목 유사도 기반 후보 나열까지만, 확정은 사람이.
- **`gameId`는 nullable** — 게임을 특정할 수 없는 이벤트(예: 종합 행사, 여러 게임 묶음 특가)도 존재하므로 필수화 금지.

### 4.3 등록 자동 채움

**결정: URL 붙여넣기 → 서버가 해당 1건만 페치해 메타데이터 추출 → 폼 프리필. 운영자·제보 폼 양쪽에 적용. 채워진 값은 전부 수정 가능.**

- 근거: 운영자 수동 등록 시간이 서비스 생존의 실질 병목. 건당 2~3분 × 주 20건의 누적 부담이 이런 서비스가 조용히 죽는 가장 흔한 이유.
- 제보 폼에도 적용 → 제보 마찰 감소 + 제보 품질 상승 → 검수 부담까지 동시 감소.
- **법적 구분**: 사용자가 지시한 단일 URL 페치는 대량 크롤링과 성격이 다름(사용자 행위의 대리 실행, 서버 부하 무시 가능). [product.md의 자동 수집 리스크](product.md)와 별개 사안으로 취급.
- 추출 대상: `og:title`, `og:image`, `og:description`, `og:url`. 플랫폼별 전용 파서는 만들지 않음(YAGNI) — OG 표준으로 커버 안 되는 값은 사람이 채움.

**필수 방어 (미인증 상태의 외부 페치 API)**
- **SSRF 차단**: 사설 IP 대역(RFC1918)·localhost·링크로컬·메타데이터 엔드포인트 거부, 리다이렉트 추적 시 매 홉 재검증, 스킴은 http/https만.
- **자원 상한**: 응답 크기 상한, 타임아웃, 리다이렉트 횟수 상한.
- **rate limit**: IP당 호출 제한 (기존 [ADR-0006](adr/0006-abuse-prevention.md)의 방어와 동일 계열).

### 4.4 데이터 신선도

**결정: 개별 이벤트가 아니라 피드 전역에 "최근 업데이트 N시간 전"을 표기.**

- 근거: 달성률·모금액을 다루지 않기로 했으므로 낡을 수 있는 값이 거의 없음. 남는 것은 "이 서비스가 관리되고 있는가"라는 신뢰 신호.
- 기각 — **카드별 갱신 시각 표기**: 표시할 변동 수치가 없어 노이즈만 늘림.
- 기각 — **운영자 일괄 갱신 화면**: 갱신할 수치가 없으니 목적 상실.
- 보조 규칙: `endAt`이 지난 이벤트는 피드에서 자동 제외(파생 상태로 계산, [ADR-0004](adr/0004-derived-status.md) 유지).

### 4.5 이미지 정책

**결정: 원본 URL 핫링크. 저장·캐싱하지 않음.**

- 근거: 저작권 리스크 최소화(복제·재배포 아님). 카드에 이미지가 있어야 시각적 위계가 서고 보드게임 서비스의 정체성이 살아남.
- 기각 — **텍스트 우선 UI**: 법적으로 가장 안전하나 보드게임 서비스로서 매력이 크게 떨어짐.
- 기각 — **운영자 직접 제작 이미지**: 저작권은 완전 회피하나 이벤트당 작업량 발생 → 커버리지와 정면 충돌.

**핫링크가 만드는 필수 요구사항 (비기능으로 승격, [§6.3](#63-이미지-내구성))**
- CDN referrer 차단 대비: `referrerpolicy="no-referrer"`.
- 로드 실패 시(원본 삭제·차단·핫링크 방지) **타이포그래피 플레이스홀더로 자동 대체**, 레이아웃 붕괴 금지.
- 이미지 유무와 무관하게 카드가 성립해야 함 = **progressive enhancement**.

### 4.6 인증과 로그인 벽

**결정: 제보는 익명 허용, 로그인은 혜택으로 유인.**

- 무인증 가능: 피드 조회, 상세 조회, 검색·필터, **제보 제출**.
- 로그인 필요: 북마크, 알림 설정, 내 제보 현황 조회.
- 로그인 혜택: 내 제보의 검수 상태 확인, 승인 알림, 기여 기록.
- 근거: 초기엔 데이터 공급이 가장 급함 → 제보에 인증 벽을 세우면 제보량이 확 줄어듦. 대신 남용 방지([ADR-0006](adr/0006-abuse-prevention.md))를 계속 유지해야 함.
- 기각 — **제보에도 로그인 요구**: 스팸은 줄지만 초기 데이터 기근을 악화. 검수 게이트가 이미 공개 피해를 1차 차단하므로 인증까지 요구할 필요 없음.

**인증 방식** — Kakao / Google OAuth2 소셜 로그인, 자체 비밀번호 미보유. **이미 구현 완료**(커밋 9597909) — 세션 쿠키 + CSRF 쿠키로 확정됐고 `/api/admin/**` 은 `hasRole("ADMIN")`으로 잠겨 있음. 현황·잔여 부채는 [spec/m3-tracking.md §0](spec/m3-tracking.md).

### 4.7 추적 (북마크·알림)

**결정: 개별 이벤트 북마크 + 마감임박 이메일 알림.**

- 기각 — **웹푸시**: email 없는 카카오 유저를 커버하고 무료지만, iOS Safari가 PWA 설치를 요구해 커버리지에 구멍. 구독 동의 UX도 추가 설계 필요.
- 기각 — **키워드/조건 구독**: 리텐션 훅으로는 가장 강하나 초기 이벤트 밀도(주 20건 수준)에선 구독 조건이 무의미하게 넓거나 좁음. 이벤트 밀도가 오르면 재검토.
- 기각 — **ICS 캘린더 연동**: 구현량 대비 효과가 좋고 "일정 서비스" 정체성과도 맞지만, **알림을 외부 캘린더에 위임하면 재방문이 발생하지 않음** → 북극성 지표와 정면 충돌. 보조 기능으로는 나중에 검토 가능.

**카카오 email 비보장 대응 (필수)**
- 카카오는 email 제공이 선택·심사 대상 → email 없는 유저 존재.
- 요구사항: **알림을 켜는 시점에 email이 없으면 그 자리에서 입력받는 플로우**. 로그인 시점이 아니라 알림 설정 시점 — 마찰을 필요한 순간으로 미룸.
- email 없이도 북마크 자체는 동작 (목록 관리 용도).

**중복 발송 방지**: `NotificationLog(user_id, event_id)` unique — 재실행·재시작에도 1회 보장. 상세는 [spec/m3-tracking.md §2](spec/m3-tracking.md).

### 4.8 검수 (운영자)

- 제보는 `PENDING`으로 생성, 운영자 승인 시 `PUBLISHED`, 반려 시 `REJECTED`. 별도 테이블 없이 `Event.moderationStatus`로 통합 — [ADR-0002](adr/0002-moderation-on-event.md) 유지.
- **검수 화면에 신규 요구사항 추가**: Game 매칭 후보 제시 및 확정([§4.2](#42-game-엔티티)), 자동 채움 값의 검증·수정([§4.3](#43-등록-자동-채움)).
- `/api/admin/**`은 ADMIN 인증으로 잠금. ADMIN은 자가가입 불가 — 허용목록 또는 DB 시드로 수동 부여.

## 5. 화면 요구사항

### 5.1 첫 화면 = 타임라인 (기간 그룹)

**결정: 기간으로 그룹핑한 세로 타임라인.**

```
─ 이번 주 ────────────────────────
 D-1   아컴호러 LCG 재판     펀딩 마감
       텀블벅 · 국내
 D-3   XX 선주문 마감        선주문 마감
─ 이번 달 ───────────────────────
 D-12  Gloomhaven 2E        펀딩 마감
       Gamefound · 해외
 D-20  보드게임 페스타       개막
       서울 코엑스 · 행사
─ 그 이후 ───────────────────────
 ...
─ 예고 (날짜 미정) ───────────────
 2026 Q4   Frosthaven 한글판
 시기 미정  XX 스튜디오 신작
```

- 근거: "언제"가 서비스의 축이므로 시간이 1급 조직 원리여야 함. 마감 압박을 가장 직접적으로 전달.
- 기각 — **카드 그리드**(현 구현): 이미지를 잘 살리고 탐색성은 좋으나 시간축이 약해져 "일정 서비스"라는 정체성이 흐려짐.
- 기각 — **월별 캘린더**: "일정을 한 곳에서"에 가장 직관적으로 들어맞아 보이지만 세 가지가 깨짐 — ① 펀딩은 기간(시작~마감)이라 점 하나로 안 찍힘, ② 날짜 미정 예고를 놓을 칸이 없음, ③ 데스크톱 우선이어도 밀도가 낮아 스크롤 대비 정보량이 적음.
- 기각 — **타임라인/그리드 토글**: 두 레이아웃을 만들고 유지해야 함. 초기 비용 2배, 전환율은 대개 낮음(YAGNI).

### 5.2 예고 섹션 분리

- **결정: 날짜가 확정된 이벤트만 타임라인 본문에, 예고는 하단 별도 섹션.**
- 스키마 영향 최소화: `datePrecision` 열거형을 만들지 않고 **`nullable startAt/endAt` + `scheduleNote` 자유 입력**으로 처리.
- 대가: 예고 섹션 내부의 정렬·필터는 포기(자유 문자열이라 정렬 불가). 예고 건수가 적을 것이므로 수용 가능한 대가로 판단. 건수가 늘면 정밀도 열거형 도입을 재검토.
- 기각 — **정밀도 필드(연/분기/월/일) 도입**: 표현·정렬 모두 정확해지지만 예고가 소수인 현 단계엔 과설계.

### 5.3 필터·정렬

**1급 축 (첫 화면 상단 노출)**
- **이벤트 유형**: 펀딩 / 선주문 / 특가 / 행사 — 유저 의도("살 것 찾기" vs "갈 곳 찾기")와 직결.
- **진행 상태**: 예정 / 진행중 / 마감임박 — "지금 후원 가능한 것만"이라는 실질 니즈. 타임라인의 기간 그룹과 일부 중복되나, 그룹은 *언제 끝나나*이고 상태는 *지금 참여 가능한가*라 서로 다름.

**2급 축 (접힘/보조 영역)**
- 지역(국내/해외) — 기본값은 반드시 **전체**. 통합이 서비스의 명분이므로 기본 분리는 자기부정.
- 플랫폼 — 플랫폼 수가 적어 초기엔 가치 낮음. 지역 필터와도 상당 부분 겹침.

**정렬**: 기본 = 마감임박순. 타임라인이 이미 시간순이므로 정렬 옵션은 최소화(YAGNI) — 추가 정렬 축은 요청이 관측되면 도입.

### 5.4 화면 목록

### 5.5 톤앤매너

**결정: 보드게임다운 따뜻함 (색·일러스트).**

- 방향: 목재·미플톤의 따뜻한 팔레트, 둥근 모서리, 일러스트 액센트. 차갑고 공학적인 대시보드 인상을 피함.
- 기각 — **정보형 대시보드(공관·고밀도)**: 데스크톱 우선·타임라인과 가장 잘 맞고 구현이 쉬우나, 보드게임의 즐거움이 전혀 전달되지 않음.
- 기각 — **중성 베이스 + 유형별 액센트색만**: 기능적으로는 우수하나 따뜻함의 요구를 충족 못 함.

**긴장 관계 (설계 시 반드시 해소)**
- 따뜻한 톤 ↔ 고밀도 타임라인은 서로 당김. 해법: **색·질감으로 따뜻함, 레이아웃으로 밀도** 를 담당시키고 여백을 과하게 쓰지 않음.
- 데스크톱 우선 ↔ 세로 타임라인은 와이드 화면에서 여백이 뜸. 해법: **행(row) 기반 고밀도 레이아웃**(좌측 썸네일 + 중앙 정보 + 우측 D-day/CTA) + **우측 사이드 영역**(필터, 예고 요약).
- 접근성: 이벤트 유형을 **색만으로 구분 금지** — 아이콘/라벨 병기 필수.

### 5.6 표기 언어

- 해외 이벤트 제목은 **원제 유지**(검색·식별 가능성). 한글명이 알려져 있으면 병기.
- Game 엔티티가 `titleKo` / `titleOriginal`을 모두 보유 → 검색은 양쪽 모두 매칭.
- UI 문구·라벨은 한국어. 서비스 자체의 다국어 지원(i18n)은 스코프 밖.

## 6. 비기능 요구사항

### 6.1 공유 미리보기 (OG) — 1급 요구사항

- **유입 채널이 커뮤니티 공유(디시·카페·디스코드)** 이므로, 링크를 붙였을 때 예쁜 카드가 뜨지 않으면 클릭 자체가 발생하지 않음 → 퍼널 진입이 막힘.
- 문제: CSR SPA는 크롤러가 JS를 실행하지 않아 OG 메타가 비어 있음.
- **요구사항**: 이벤트 상세·게임 페이지 URL에 대해 **크롤러 User-Agent 요청 시 서버가 OG 메타를 채운 HTML을 응답**(프리렌더). 일반 사용자 요청은 기존 SPA 그대로.
- OG 이미지: 원본 이벤트 이미지 URL을 그대로 지정(핫링크 정책과 동일).
- 이로써 **[ADR-0001](adr/0001-tech-stack.md)의 CSR SPA 선택은 유지**됨 — 검색 유입이 주력이 아니므로 SSR 전면 전환은 불필요. 대신 OG 프리렌더를 조건으로 붙임.

### 6.2 성능·규모

- **목표(측정 대상)**: 누적 이벤트 수만 건, 동시 접속 수백 명 상황을 견디는 것을 목표로 함.
- **구현 방침**: 목표를 문서에 명시하되 **캐시 레이어·이미지 프록시·읽기 복제는 지금 만들지 않음.** 아키텍처에 삽입 지점만 열어두고, 실제 도입은 성능 저하가 관측된 뒤.
  - 근거: 검증되지 않은 트래픽에 대한 선제 최적화는 복잡도만 남김(YAGNI). "성장을 고려했으나 근거 없이 만들지 않았다"가 더 방어 가능한 판단.
- **응답 예산**: 피드 p95 ≤ 500ms, 상세 p95 ≤ 300ms.
- **쿼리 규칙**: 반복문 내 쿼리 금지, 배치 로딩 + `groupBy`/`associateBy`. 타임라인은 Game·유형별 필드를 함께 로드하므로 **N+1 위험이 특히 높은 지점**.
- **인덱스**: `(moderation_status, end_at)` 유지. Game 조인 경로 인덱스 추가 검토.

### 6.3 이미지 내구성

- `referrerpolicy="no-referrer"` 적용.
- 로드 실패 시 타이포그래피 플레이스홀더로 대체 (게임/이벤트 제목 기반), **레이아웃 시프트 없음**.
- `loading="lazy"`, 고정 aspect-ratio로 CLS 방지.
- 이미지가 전부 실패해도 피드가 온전히 기능해야 함.

### 6.4 반응형·접근성

- **데스크톱 우선 설계**, 단 모바일 대응은 타협 불가 — 커뮤니티 공유 링크는 실제로 모바일에서 열림.
- 브레이크포인트: 데스크톱(사이드 영역 노출) / 태블릿(사이드 접힘) / 모바일(단일 컬럼, 필터는 시트).
- 접근성: 명도 대비 4.5:1 이상, 색만으로 정보 전달 금지, 키보드 내비게이션, 이미지 `alt`.

### 6.5 보안

- URL 자동 채움 API의 SSRF 방어·자원 상한·rate limit ([§4.3](#43-등록-자동-채움)).
- 익명 제보 남용 방지: honeypot + IP rate limit ([ADR-0006](adr/0006-abuse-prevention.md)).
- `/api/admin/**` ADMIN 인증, ADMIN 자가가입 불가.
- 타 유저 자원 접근 차단: 북마크·제보 조회는 소유자 검증.
- 현재 유저는 `SecurityContext` 직접 접근 금지 → `CurrentUserProvider`류 주입.

### 6.6 관측성

- 원본 링크 클릭 이벤트 기록 (KPI 선행지표의 데이터 소스).
- 피드 전역 갱신 시각 산출 ([§4.4](#44-데이터-신선도)).
- 운영자 등록 소요 시간 측정 (자동 채움 효과 검증).

## 7. 로드맵 (전면 재수립)

> 기존 Phase 1/2 구분은 이번 인터뷰 결과로 **폐기**. 퍼널 단계에 맞춰 재정의.

### M0 — 재설계 기반

- 데이터 모델 확정: `Game` 엔티티 신설, `EventType`에 `OFFLINE_EVENT` 추가, 유형별 필드 정리, `scheduleNote` 추가, `goalAmount`/`currentAmount`/`currency` 및 `fundingProgressPercent()` 제거.
- `EventStatus`에 `ANNOUNCED` 추가 — `startAt`·`endAt`은 **이미 nullable이지만** 둘 다 null이면 `ONGOING`으로 떨어지는 현 동작이 예고 이벤트를 "진행중"으로 표시함([ADR-0004](adr/0004-derived-status.md) 개정).
- Flyway 마이그레이션 작성. 기존 데이터 이관 방침 결정.
- 문서 정합화는 완료됨([§10](#10-기존-문서-정합성)) — ADR 0007~0010 신설, 0001·0003·0004·0006 개정.

### M1 — 발견 (Discovery)

- S1 타임라인 피드 (기간 그룹 + 예고 섹션).
- 1급 필터(유형·상태), 2급 필터(지역·플랫폼).
- S2 이벤트 상세, S3 게임 페이지.
- **OG 프리렌더** ([§6.1](#61-공유-미리보기-og--1급-요구사항)) — 유입 채널의 전제 조건이므로 M1에 포함.
- S6 운영자 등록 + **URL 자동 채움**.
- 이미지 핫링크 + fallback.
- 톤앤매너 적용된 디자인 시스템 1차.

**M1 완료 기준**
- 운영자 혼자 활성 이벤트 20~30건을 유지할 수 있음.
- 커뮤니티에 링크를 공유하면 카드가 정상 노출됨.
- **행사·예고가 각각 최소 3건 이상 게시돼 있음** — 미달 시 정체성 재정의가 실패한 것이므로, 범위를 펀딩·선주문 중심으로 되돌릴지 재판단([§1](#1-서비스-정체성) 리스크).

### M2 — 공급 (Supply)

- S4 익명 제보 폼 + URL 자동 채움 + honeypot/rate limit.
- S5 검수 큐 + Game 매칭 확정 UI.
- 제보 승인율 측정.

### M3 — 추적 (Retention)

- Kakao/Google OAuth2 로그인, `/api/admin/**` 잠금.
- S7 북마크 + 알림 설정 (email 미보유 시 입력 플로우).
- 마감임박 이메일 알림 배치 + `NotificationLog` 멱등 게이트.
- 로그인 혜택: 내 제보 현황·승인 알림.

**M3 완료 기준**: 북극성 지표(주간 재방문)를 유저 단위로 측정 가능.

### M4+ — 장기 비전

## 8. 장기 비전

> "언젠가"로 기록. 각 항목에 **진입 조건**을 명시해 조기 착수를 방지.

### 8.1 자동 수집

- 목표: 운영자 수작업 의존을 끊어 커버리지와 지속가능성 확보. 서비스의 핵심 병목.
- **합법 경로는 Gamefound 공식 API 뿐** — 텀블벅·Kickstarter는 ToS가 자동 수집을 명시 금지, 보드라이프는 도구명 직접 차단으로 거부 의사 문서화. 우회는 법적·포트폴리오 양면에서 부적절. 상세 조사·순회 주기 스케치는 [product.md](product.md) 유지.
- **진입 조건**: M2 완료 후에도 커버리지가 목표(활성 20~30건)에 미달하거나, 운영자 등록 시간이 지속 불가능한 수준으로 관측될 때.

### 8.2 게임 단위 생애주기 추적

- 목표: "펀딩 → 제작 → 배송 → 국내 정발 → 확장팩"을 한 게임 페이지에서 추적. "내가 후원한 게 언제 오나"에 답함.
- Game 엔티티([§4.2](#42-game-엔티티))가 이미 선결 조건을 충족 → S3 게임 페이지가 그 씨앗.
- **진입 조건**: 게임당 평균 연결 이벤트 수가 2건 이상으로 축적될 때 (그 전엔 페이지에 보여줄 것이 없음).

### 8.3 커뮤니티 (평점·댓글·공동구매)

- 목표: 체류 시간과 재방문 증대.
- **비판적 유보 (기록)**: ① 국내에서 게시판을 열면 정보통신망법상 운영자 의무(신고 처리·임시조치)가 발생 — 1인 운영에 실질 부담. ② "집계 도구"라는 정체성을 흐림. ③ 초기 트래픽에선 빈 댓글창이 오히려 서비스가 죽어 보이게 함.
- **진입 조건**: 북극성 지표가 안정적으로 성장하고, 사용자로부터 커뮤니티 기능 요구가 실제로 관측될 때. 도입 시에도 **평점부터** (운영 부담이 가장 낮은 순서).

### 8.4 (참고) 기각 — 가격 비교·최저가 추적

- 상업적 가치는 가장 크나 데이터 수집 난이도가 최상(가격·환율·배송비·관세). 자동 수집조차 미해결인 단계에서 착수 불가. 인터뷰에서도 선택되지 않음.
