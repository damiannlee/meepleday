# 제품 계획 — 성공지표 · 데이터 수급

> 서비스 개요는 [README](../README.md), 설계 근거는 [docs/adr](adr). 이 문서는 "무엇으로 성공을 판단하고, 데이터를 어떻게 채우는가".

## 성공지표 (KPI)

> 초기엔 트래픽이 작아 정량 지표가 노이즈 → **정성 신호 우선**, 정량은 방향 확인용.
> 공개 출시는 완성형(인증+마감임박 알림)으로 하므로([ADR-0005](adr/0005-deployment-target.md)) 재방문을 출시 시점부터 구동(알림 훅)·정밀 측정(유저 단위) 가능.

**북극성 지표**
- **주간 재방문 유저** — 다시 온다 = "마감 추적 도구"로 자리잡음. 지속 가치를 반영해 일회성 방문보다 서비스 본질에 정렬.

**선행지표 (방문당 가치)**
- 원본 링크 클릭률 — 방문이 실제 가치("이벤트 발견 후 이동")를 줬는지. 재방문의 선행 신호.

**공급 건강도**
- 게시된 이벤트 수(국내/해외 비율) — 커버리지, "볼 게 있는" 상태인가.
- 제보 승인율 = PUBLISHED / (PUBLISHED+REJECTED) — 크라우드소싱 품질.
- (Phase 2) 마감임박 알림 클릭률 — 재방문 훅의 직접 효과.

**초기 정성 목표**
- 실사용 보드게이머 5명이 "국내+해외를 한 곳에서 보니 유용하다" 확인 → PMF 초기 신호.

## 데이터 수급 전략

> 크롤러는 후순위([현재 위치](../CLAUDE.md)). 초기 공급은 운영자 수동 + 사용자 제보 두 갈래([ADR-0002](adr/0002-moderation-on-event.md)).

**1단계 — 운영자 시딩(출시 시)**
- 운영자(본인)가 주기적으로 신규 이벤트 등록.
  - 국내: 텀블벅 보드게임 카테고리, 보드라이프 신작/펀딩.
  - 해외: Kickstarter·Gamefound 보드게임 카테고리 주요 펀딩.
  - 이노펀딩은 제외 — 도메인 DNS 소실(NXDOMAIN, 2026-07 확인)로 서비스 존속 불명.
- 목표: 상시 게시 이벤트가 "볼 만한" 최소량(예: 활성 20~30건) 유지.

**2단계 — 크라우드소싱 제보**
- 사용자 제보로 롱테일(소규모 팀·마이너 플랫폼) 보강. 검수로 품질 담보.
- 제보 남용 대비는 [ADR-0006](adr/0006-abuse-prevention.md).

**3단계 — 자동 수집(Phase 3+, defer)**
- 크롤러/스크래퍼 또는 공식 API(있는 경우)로 수집 자동화.
- **선결 과제**: 각 플랫폼 이용약관·`robots.txt`·저작권(이미지·설명 재사용) 법적 검토. 무단 대량 수집은 리스크 → 착수 전 별도 판단 필요.
- 아래 두 절은 **착수 시점 재검증 대상 스케치** — 조사 시점 2026-07, ADR 승격 전 단계.

### 플랫폼별 수집 가능성 (2026-07 조사)

| 플랫폼 | ToS 자동수집 금지 | robots.txt | 공식 API | 판단 |
|---|---|---|---|---|
| Gamefound | 명시 조항 미발견 | 개방(`Allow: /`) | **있음** — `getActiveCrowdfundingProjects` 등 3개, 인증 불필요 | **유일한 합법 경로**, 착수 시 여기부터 |
| 텀블벅 | **있음** — 이용약관 제38조 1항 8호 "자동화된 수단" 금지 | 일부 경로만 Disallow | 없음 | 크롤링 불가, 제휴 문의 경로만 |
| Kickstarter | **있음** — crawl/spider/scraping/data mining 명시 금지 | Cloudflare가 비-브라우저 요청 차단(실측 403) | 없음(승인제·사실상 불가) | 대상 제외 |
| 보드라이프 | ToS엔 없음 | `curl`·`Scrapy` 등 **도구명 직접 차단** = 거부 의사 문서화 | 없음 | 회색지대, 사실상 불가로 취급 |

- 공통 원칙: **`robots.txt` 준수 ≠ 법적 안전**. 위법성은 ToS 명시 금지 문구와 접근 통제(로그인·anti-bot) 우회 여부로 갈림.
- ToS가 막은 플랫폼은 우회 대신 **제휴·API 협의**가 유일한 정공법. 우회 구현은 포트폴리오 리스크로도 부적절.

### 순회 주기 설계 (스케치)

- **전제 재정의**: 인기 캠페인은 5~18분에 펀딩 완료 → 애그리게이터가 속도로 이길 수 없음. 커뮤니티 소비 주기도 주간~반기(BGG geeklist 등)로 느슨.
- → 주기 목표는 latency("얼마나 빨리 알리나")가 아니라 **coverage("놓치는 이벤트가 없나")**.

| 티어 | 대상 | 주기 | 근거 |
|---|---|---|---|
| 신규 발견 | 활성 캠페인 목록 diff | 평시 1h, 화 21:00~03:00 KST 15m | 론칭이 화요일 오전 ET 집중(342K 캠페인 분석) |
| 마감 임박 | `ENDING_SOON` 구간 | 15m | 마지막 3일에 자금 20% 집중, 유저 행동 압박 최대 |
| 정보 갱신 | 중간 구간 캠페인 | 6~12h | 중간 구간(42.6%) 완만·평탄, 금액 오차는 유저 손해와 무관 |

- 벤치마크: **Kicktraq**(동일 문제 실운영) = 매시간 + 마감 48h 이내 15분. 이 48h는 [ADR-0004](adr/0004-derived-status.md) `ENDING_SOON` 임계값과 일치 → **고빈도 티어는 별도 상수 대신 `ENDING_SOON` 파생 상태 재사용**(임계값 단일 소스 유지).
- Gamefound API 응답은 *"cached for a short duration"*(TTL 미공개) → **15분을 폴링 하한**. 실제 `Cache-Control`/`ETag` 실측 후 사후 조정.
- 기각 — **adaptive scheduling**(변경률 기반 소스별 최적 주기): 소스 수천 개일 때 유효. 소스 2~3개 규모엔 과설계(YAGNI). 고정 tiered로 충분.
- 기각 — **early bird 전용 고빈도 티어**: 얼리버드 창(24~48h)은 시간당 폴링으로 커버되고, 소진 시간 정량 분포 자료 자체가 없어 근거 부재.
- **자료 없어 미확정**: 얼리버드 소진 분포 / 커뮤니티 확산 속도 / 알림 지연-가치 벤치마크 / Gamefound 캐시 TTL. 추측 대신 **첫 폴링 로그로 실제 변화 간격 자체 계측** 후 주기 조정.

## 스코프 밖 (의도적)

- 결제·펀딩 중개(우린 **집계·안내**만, 거래는 원본 플랫폼으로 이동).
- 리뷰·커뮤니티·평점 — 핵심 가치(이벤트 집계) 검증 후에나 검토.
